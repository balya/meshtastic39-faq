# Meshtastic FAQ Documentation Agent

## Цель проекта
Проект содержит базу знаний и FAQ по Meshtastic для публикации в Telegram-группе Калининградского сообщества.
Каждая статья представляет собой ответ на один конкретный вопрос, который регулярно возникает у участников сообщества.
Основная задача — создавать точные, актуальные и практически полезные ответы на реальные вопросы пользователей Meshtastic.

Статьи должны:
- отвечать на один конкретный вопрос;
- содержать короткий, но достаточно полный ответ;
- быть удобными для чтения в Telegram;
- хорошо находиться через поиск Telegram;
- содержать только проверенную информацию;
- основываться на официальной документации Meshtastic и устоявшейся практике сообщества;
- быть понятными новичкам и одновременно технически корректными.

При написании статьи следует стремиться к тому, чтобы пользователь получил ответ на свой вопрос максимально быстро, без необходимости изучать дополнительную документацию или длинные обсуждения.
Все ответы должны быть лаконичными, легко читаемыми, содержать минимум воды и максимум полезной информации по существу вопроса.
Основной приоритет — точность, актуальность и практическая польза информации.

## Источники информации
При подготовке статьи необходимо использовать источники в следующем порядке приоритета.

### 1. knowledge.md
Основной источник знаний проекта.
Содержит:
- принятые в сообществе рекомендации;
- локальные соглашения и правила;
- проверенные настройки;
- полезные ссылки;
- разъяснения спорных вопросов;
- информацию, многократно подтверждённую практикой сообщества.

Если информация в knowledge.md противоречит актуальной официальной документации Meshtastic, необходимо сообщить об этом пользователю и использовать данные из официальной документации.

### 2. Официальная документация Meshtastic
Используется для проверки актуальности информации, технических деталей и изменений в новых версиях прошивки.

### 3. Официальные репозитории и документация производителей оборудования
Используются для проверки характеристик устройств, схем подключения и аппаратных ограничений.

### 4. Проверенные материалы сообщества
Используются только в случае отсутствия необходимой информации в предыдущих источниках.


Если информации недостаточно или есть сомнения в её актуальности, необходимо выполнить дополнительный поиск в интернете.

При противоречии между источниками необходимо:
1. Проверить актуальность информации.
2. Отдать предпочтение официальной документации.
3. Если сообщество сознательно использует иной подход, указать это явно в статье.

Не использовать неподтверждённую информацию, предположения или личные догадки как факты.

Если по вопросу не удалось найти достаточного количества достоверной информации, необходимо явно указать это в статье. Не следует заполнять пробелы предположениями.

## Работа с вопросами
Список вопросов находится в файле questions.md.

Вопросы хранятся в формате:
- [x] 0001
  Что такое Meshtastic?
- [ ] 0002
  Как настроить MQTT?
- [ ] 0003
  С чего начать новичку?

Числовой идентификатор является уникальным идентификатором статьи и не должен изменяться.

Для каждого вопроса существует отдельный файл статьи:  
articles/0001.md  
articles/0002.md  
articles/0003.md  

Статус вопроса определяется только по галочке в questions.md.

Вопросы с пустой галочкой ([ ]) считаются находящимися в работе. Соответствующие статьи считаются черновиками и могут свободно создаваться, изменяться и дополняться агентом.

Вопросы с установленной галочкой ([x]) считаются опубликованными. Соответствующие статьи считаются проверенными и не должны изменяться агентом без явного указания пользователя.

Перед созданием новой статьи или изменением существующей агент должен определить идентификатор вопроса в questions.md и работать со статьёй, соответствующей этому идентификатору.

Агент не должен самостоятельно изменять файл questions.md, создавать новые идентификаторы или менять существующие идентификаторы вопросов.

Если для вопроса отсутствует файл статьи, агент может создать его.
Если агент обнаружит проблемы в структуре базы знаний (например дублирующиеся идентификаторы, несоответствие вопроса и идентификатора или другие нарушения структуры), он должен сообщить об этом пользователю в чате. Агент не должен самостоятельно исправлять подобные проблемы, если это не было явно запрошено пользователем.

## Работа со статьями
Статьи хранятся в каталоге articles.
Каждая статья отвечает на один конкретный вопрос из файла questions.md.

Имя файла статьи должно совпадать с идентификатором вопроса:  
articles/0001.md  
articles/0002.md  
articles/0003.md  

В начале каждой статьи должны присутствовать метаданные:
```
---
id: 0001
question: Что такое Meshtastic?
---
```

Значение id должно совпадать с идентификатором вопроса в questions.md.
Значение question должно содержать актуальную формулировку вопроса.
Статья должна отвечать только на вопрос, указанный в поле question.

При необходимости разбиения темы на несколько статей следует сообщить об этом пользователю.

Статья должна содержать краткий и понятный ответ на вопрос, практические рекомендации и достаточное количество информации для решения задачи пользователя без необходимости изучать дополнительные материалы.

Все ответы должны быть лаконичными, легко читаемыми и содержать минимум воды.
При использовании терминов из keywords.md необходимо оформлять их как хештеги согласно правилам проекта.

Перед изменением статьи агент обязан определить её статус через соответствующий вопрос в `questions.md`.

## Формат статей
Каждая статья представляет собой ответ на один конкретный вопрос.

Основная цель статьи — быстро дать пользователю понимание сути вопроса и помочь принять правильное решение или выбрать дальнейшее направление изучения темы.

Ответ должен быть:
- лаконичным;
- технически точным;
- легко читаемым;
- практически полезным;
- основанным на документации и проверенной информации.

Предпочтительный объём статьи — до 1000 символов.
В случаях, когда вопрос требует дополнительных пояснений, допускается увеличение объёма до 2000 символов.

Не следует пытаться подробно раскрывать все аспекты темы или заменять статьёй официальную документацию.
Если для полного понимания вопроса требуется дополнительная информация, рекомендуется указать ссылки на связанные статьи базы знаний, официальную документацию или другие полезные источники.

Допускается превышение рекомендуемого объёма для списков настроек, таблиц, примеров конфигурации и другой справочной информации, если это необходимо для корректного ответа на вопрос.

Избегать:
- длинных вступлений;
- повторов;
- избыточных технических деталей;
- теоретических рассуждений, не влияющих на практическое решение вопроса.

Статья должна быть настолько короткой, насколько это возможно, но при этом полностью отвечать на поставленный вопрос.

## Форматирование

Статьи предназначены для публикации в Telegram и должны использовать только поддерживаемые Telegram элементы форматирования.

Допускается использование:  
**жирный текст**  
__курсивный текст__  
~~перечеркнутый текст~~  
`моноширный текст`  
||текст под спойлером||  

Блоков кода:
```
блок кода
```

Для улучшения читаемости рекомендуется использовать:
- короткие абзацы;
- маркированные списки;
- нумерованные списки;
- выделение важных мыслей жирным текстом.

Не следует злоупотреблять форматированием. Выделять следует только действительно важную информацию.

Настройки, команды, адреса серверов, MQTT-топики, пути к файлам, имена параметров и другие технические значения рекомендуется оформлять моноширинным текстом.

Примеры:
Адрес: `mqtt.onemesh.ru`  
Топик: `msh/RU/KGD`  
Параметр: `hop_limit`  
Команда: `meshtastic --info`  

Если требуется привести пример конфигурации, команды или фрагмента кода, следует использовать блок кода.

Ссылки необходимо указывать обычным текстом без дополнительного форматирования. Пример:
Ссылка: https://meshtastic.org/docs/introduction/

Не использовать:
- HTML-разметку;
- Markdown-ссылки вида [текст](ссылка);
- таблицы;
- изображения;
- элементы форматирования, не поддерживаемые Telegram.

## Хештеги
Для организации навигации и поиска внутри Telegram используется файл `keywords.md`.
Файл содержит список ключевых слов, которые должны автоматически превращаться в хештеги при использовании в статьях.

Пример содержимого `keywords.md`:
```text
- mqtt
- антенна
- gps
- bluetooth
- heltec
```

Если в статье встречается слово из `keywords.md`, перед ним необходимо добавить символ `#`. Примеры:
```
Основные настройки #mqtt

Выбор #антенны для городской ноды

Как работает #gps в Meshtastic
```

Хештеги являются частью обычного текста статьи и не должны выноситься в отдельный список в начале или конце статьи.

Допускается использование различных форм слова, если они естественно встречаются в тексте. Пример:

```text
Выбор #антенны для городской ноды
```

Не следует добавлять одинаковый хештег более одного раза в рамках одной статьи. Если ключевое слово встречается несколько раз, хештег ставится только при первом упоминании.

Хештеги используются для сквозной навигации по базе знаний и быстрого поиска связанных материалов через поиск Telegram.
